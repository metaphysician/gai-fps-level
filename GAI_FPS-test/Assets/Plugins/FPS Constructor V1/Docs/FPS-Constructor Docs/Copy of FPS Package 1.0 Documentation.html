Package Description and Contents

Thank you for interest in the Dastardly Banana FPS Constructor package available on the Unity Asset Store. This Unity Package contains everything you need to accelerate the development of your First Person Shooter game, including:

  • A simple framework for making all variety of FPS weapons in minutes, from the mundane to the extraordinary.
  • An easy way to integrate weapon models and animations - no coding required!
  • A system for environmental effects like bullet penetration, impact particles and sound effects.
  • Pre-made example weapons, including seeking missiles and an orbital laser designator.
  • Easy to configure smart cross-hairs, enemy damage, and weapon pickup.
  • And much more.

This package includes the following files:

  • A Unity 3.x package containing a configured demo scene and all assets.
  • This documentation in PDF and HTML

The most current version of this documentation can be found at http://www.dastardlybanana.com/Docs/FPSConstructor/

Support Forums for the Package are at

http://www.datardlybanana.com/forums

The package contains one script  and some particles in the Standard Assets folder so you'll need to import them as well.  Don't worry, none of your standard assets will be overwritten. It was necessary to put MouseLookDB in the Standard Assets folder b ecause of Unity's order of compilation

Exploring The Demo Scene

The demo scene is a good way to explore the functionality of the package and to understand 

how the provided components can be used to create a game. You can navigate around a simple shooting range, fire at targets, switch between pre-configured weapons and see a sample of the assorted effects supported by the package, including:

  • A selection of different weapon types
  • Changing gun positions and motion when walking, running, crouching,aiming down sights, reloading, etc.
  • Footstep sounds as you walk or run different materials
  • Different particle effects, decals and sounds when you shoot different materials.
  • Force applied to objects when hit
  • Applying damage to targets
  • Bullet penetration
  • Simple ammo and health GUI displays
  • Weapon selection
  • Weapon Pickup

Demo Controls

  • Number Keys - Select Weapon Slot
  • WSAD for movement
  • C - Crouch
  • R - Reload
  • Left Click - Shoot
  • Right Click - Aim Down Sights
  • Spacebar - Jump
  • Shift - Sprint
  • Tab - Pickup

The Demo Controls are set in Edit->Project Settings->Input.

Before getting started we recommend that you duplicate the Simple Scene so you have a copy to experiment with. Making changes to the demo scene is probably the easiest way to learn how the package works.

The Demo Scene Hierarchy

The basic hierarchy of objects in the demo scene is shown to the right. The scene contains some crates, targets, exploding barrels, a wall and a pickup weapon which are used to showcase the capabilities of the Dastardly Banana FPS Constructor package. The most important components the Manager and Player Objects.

  • Manager - The Manager object contains scripts to 
    manage the ammo, to create effects sets (sounds, decals, etc), and to apply the effects automatically to game objects. 
  • Player - The Player hierarchy contains the core GameObjects and scripts for the FPS package. This includes the cameras, the weapons and the main scripts that control the player and weapons.

We'll be reviewing each of the objects and scripts in detail later this documentation. 

The Player Hierarchy

The following components are under the Player GameObject.

  • Player: The Player contains scripts to control motion and Input (CharacterController, MouseLook, CharacterMotor, FPSInputController) as well as the Footstep sound, PlayerHealth and the sample HealthDisplay GUI script.
  • The Weapon Camera: Two cameras are used to prevent the guns from being clipped. The Weapon Camera is set to draw only the player layer. The Weapon Camera also has PlayerWeapons Script which holds the weapons that are selectable during the game, some common animations, and a script to execute the "Death Effects" when the Player dies. 
  • Optional Crosshairs: Objects that have optional crosshair textures that can be used by the weapons. The demo scene has two optional crosshair textures, CircleCrosshair and LauncherCrosshair.
  • GunMotion: This is the parent object for all weapons that are available in the scene. GunMotion has scripts to control the visual sway of the guns as well as the jumping and crouching effects.
  • HitDirectional: An indicator used when a player takes damage to show what direction the damage came from. 
  • The Weapons: The Weapons are attached to the GunMotion object. The Demo Scene comes with 10 sample weapons (FAL, Flamer, M79, M1911, Orbital Designator, Plasma Cannon, Seeker Launcher and Spas-12.)
  • MainCam:  The camera used to render the scene. It also contains a script used to render death effects.

The Weapons

The weapons are attached to the GunMotion Object. The package includes the following 10 sample weapons that demonstrate the capabilities of the system. You may use these weapon models in a game or configure your own models as described later in this documentation.

  • Bearded Axe - The Bearded Axe is an example of a Melee weapon that does damage on contact with an enemy. An example of a hitscan weapon.
  • FAL - Automatic assualt rifle with underslung grenade launcher. This is a good example of the use of a secondary weapon.
  • Flamer - A Flamethrower that projects a stream of flammable liquid. An example of a spray weapon.
  • M1911 - A /45 caliber handgun.
  • MAS-36 - A powerful bolt-action rifle.
  • Orbital Designator - Calls down an orbital laser strike on a selected target. An example of  the power and flexibility of launcher weapons.
  • Plasma Cannon - A launcher weapons that fires a charged plasma projectile. 
  • Seeker Launcher - A launcher weapon that Locks on to up to three targets and fires Seeker Missiles.
  • Spas 12 - A pump action shotgun.
  • Thumper - A Single-shot grenade launcher.
  • Null- A placeholder required for empty weapon slots.

Each weapon includes the following scripts attached at the top level. A prefab is provided PefabsDB-> Setup folder with the required scripts pre-attached.

  • Gun Script  - this is the brains of the weapon and is described below.As an example we'll take a quick look at the configuration of the M1911. The M1911 object has the following scripts attached.
  • Ammo Display - This script displays the Ammo information in the GUI. It can be used as an example for a more sophisticated UI.
  • Audio Source - Used to play the sound specified in the Gun Script.
  • Weapon Locking - Used in the Dastardly Banana Upgrades and Store package. This package is not currently compatible with the FPS Constructor 1.0 package but we intend to upgrade it in the future.
  • Weapon Info -  Also used in the Dastardly Banana Upgrades and Store package
  • AimObject - The parent for the weapon model, the AimObject controls the position and visual motions of the weapon when aiming, sprinting, etc. It also controls how long you can sprint with that weapon and what the recovery time is. The weapon models are described later in this document.
  • Weapon Display - A sample script to display weapon info when a new weapon is selected.

The Gun Script

The Gun Script is the brains of the weapon and by changing properties you can change the behavior of all apects of the weapon. Most properties have tool-tips so you can quickly learn what behavior the property controls.

As an example we'll look at the M1911 which is already equipped in slot 1. The Gun Type is set to hitscan which means that when the weapon is fired a raycast is used to determine where the hit point is and the effects are applied. The Gun Type drop-down can be used to change the weapon to a launcher, melee or spray weapon and as the type changes the custom inspector shows only the properties that are relevant for that type of weapon.

These weapon types are described in detail below. For now we'll leave the M1911 as a hitscan weapon.

To get a better understanding of the effects of the configurable properties, run the scene and then make some changes. For instance, increase the force in the Shot Properties section from 200 to 10000 and shoot some boxes. Note how they fly away! Try changing the burst count to three and making the gun less accurate.  You should also explore the other example weapons to see the behavior of launcher, hitscan and melee weapons.

Player Weapons Script

As you play Demo Scene you can change weapons using the number keys 1 through 9..The equipped weapons are controlled by the Player Weapons script attached to the Weapons Camera. As you change weapons note that the Selected Weapons value changes. You can change the slot assignments by dragging the weapons under GunMotion into the slots. Note that you can't have the same weapon in two different slot.

You can increase or decrease the number of weapons slots by changing the Size property. You should drag the provided Null weapon into any unused slots. We'll show you how to pickup new weapons and change weapons within a slot later in this document.

Also note that you can set inverted y controls on Player Weapons.

Picking up and Equipping Weapons

The demo scene includes one "pick-uppable" weapons, the FAL. The pick-uppable weapons are simply models with the SelectableWeapon Script attached. The FAL pickup also has the script HighlightFX attached which displays a GUI element when the weapon is highlighted. Prefabs for the Pickup/Drop weapon models are in the Demo Assets project Folder.

 

To add a pickup weapon to the scene follow these steps.

  1. Place a model in the scene and add a collider to to the model.
  2. Drag an actual weapon into the 'weapon slot' of the SelectableWeapon script in the inspector.
  3. Optionally attach a HighlightFX script (more below)

Picking up the Weapons is actually controlled by the PickupWeapon script attached to the Weapon Camera. This script allows you to set the maximum distance for pickup, allows you to 

discard the current weapon and destroy the Pickup object in the scene. 

The model discarded is set in the WeaponInfo script attached to the weapon. 

The HighlightsFX script is an example implementation of one of the package extension points and can be found in the ScriptsDB->Editable project folder. The script implements two methods, HighlightOn() and HighlightOff() which are called when the SelectableWeapon script send HighlightOn and HighlightOff message.

The Manager Object

The FPS package provides an easy way to manage effects sets and to consistently apply them to objects in your scene. This capability is configured by two scripts attached to the Manager prefab.

  • Effects Manager -  The Effects Manager allows you to create effects sets representing different materials in your game. The effects set defines the bullet decals and dents to be applied when a material is shot, the hit particles, bullet sounds, collision sounds and footstep sounds. 
  • The Effects and Penetration Wizard - This wizard finds all colliders in the scene, groups them by name, and allows you to select the Effects Set to be used. It also allows you to control bullet penetration.

The Effects Manager

The Effects Manager Editor allows you to create and configure Effects Sets. To edit an existing set simply click on the drop-down for Select Sets to Edit and select the set. To create new set simply select New Set in the drop-down and then Change the Effects Set Name.

Each of the effect types (Bullet Decals, Dent Decals, etc) are arrays which allow you to add one or more objects of the appropriate type (as you add objects to an array, the array automatically expands) The system rotates through the specified objects allowing you to create variation in your scene.

The Effects and Penetration Wizard

The Effects and Penetration Wizard allows you apply an Effects Set to a group of objects in your scene and to specify bullet penetration properties. It does this by finding all colliders in the scene and grouping them by name. You can switch between applying the Effects Sets and Bullet Penetration using the Effect dropdown.

When UseEffects is selected you will see a list of Colliders names found in the scene. If multiple colliders are found with the same name they are grouped together and the same effects are applied to the entire group. If Enable is checked the script UseEffects.js is attached to the colliders and the chosen Effect Set is enabled. If enable is unchecked the UseEffects.js script is removed from the objects.

When BulletPenetration is selected you see the same list of colliders but now you can set the Resistance to bullet penetration. If Enable is checked, the script BulletPenetration.js is attached to the object and bullet penetration is enabled.   If the gun's Penetrate Value is greater than or equal to the shot object's Penetration Level, then the bullet will keep going. Furthermore, when a bullet goes through an object, its Penetrate Value is reduced - so a shot from a gun with a Penetration value of 2 could pass through two objects with a Penetration Level of 1, but could not pass through a third.

To get a better understanding of how the effects manager works you can add a new collider into the scene and try applying different effects sets and Resistance Levels. You can also create a new Effects Set and experiment with the different settings.

Applying Damage

The demo scene includes two exploding barrels which demonstrate how you can apply damage to an enemy or any object in the scene. In the default setup you can shoot a barrel three times with the M1911 before it explodes. This behavior is controlled by the EnemyDamageReceiver script attached to the barrels.

Hit Points is the health of the object. When it reaches 0 the death effect is played and the object is replaced with the Dead Replacement. This script is described in detail later in this document

Setting Up Your Own Scene

This section provides high-level instructions for integrating the FPS Package into an existing or new scene.

  1. Drag in the Manager Prefab from PrefabsDB->Setup. If a First Person Controller or Camera is already in the scene, delete it.


  2. Drag in the Player Prefab from PrefabsDB->Setup
  3. Deactivate the CircleCrosshair and LauncherCrosshair objects under the WeaponCamera.
  4. Add you weapons (or some of the sample weapon prefabs) to GunMotion under Weapon Camera. Tweak the weapon configurations. Make sure you drag the prefab and not the source since the source contains the models not the weapon!


  5. Assign any weapons from step 3 to slots by dragging them into the Weapons array of WeaponCamera->PlayerWeapons


  6. Create Effects Sets and apply to materials in your scene as described in the section Configuring the Rest of Your Game and The Manager Object
  7. Add the EnemyDamageReceiver Script to any objects you want to do damage to.

Configuring the Player

Movement Speed and Sprint (1.1.2+)

There is a new Movement Values Component on the Player, which allows you to more easily change the movement speed of the player. It also has global values for sprint (how long you can sprint, etc). These values an be overwritten for an individual weapon by checking Override Sprint on its AimMode.

Camera Smoothing (1.1.2+)

As of 1.1.2, the FPS Constructor has optional mouse smoothing. To deactivate it, go to the MouseLookDBJS component on both the Player and the Weapon Camera, and uncheck Smooth. You can also change the smoothing amount by editing Smooth Factor.

Prone (1.1.2+)

Prone is an additional level of crouch in which the player lies on the ground. You can deactivate it by un-checking Use Prone in the Character Motor on the Player. There is also a new type of footstep sound called a crawl sound which is used for prone player. This can be changed in the effects manager just like the standard footstep sounds (by default there are no crawl sounds defined, meaning that the player moves silently).

Making and Configuring Weapons

Adding a Weapon to the Scene        

Drag the weapon prefab onto GunMotion on your player (under the weapon camera). Then drag your animated weapon model onto the aimobject. Add the gun child animation script onto your model, and change the names in the script to match the names of your animations (also keep in mind that every animation other than fire and reload is optional and can be left blank if there is no animation for it). Then configure the aim, hip, and sprint positions of your weapon. now you're ready to start configuring it's behavior!

The Weapon Types

Every weapon created with the package has a weapon type. Weapon types are broad categories that determine the very basics of how the weapon functions. The options available in customizing your weapon, and the way you will set up your weapon in the scene, will vary depending on what type you have selected. The supported types are described below.

Hitscan: Hitscan weapons are the most basic form of weapon. A hitscan weapon, when fired,

checks to see what the player is aiming at, and, if the target is in range and is a valid target, instantly applies its damage and effects to that object. A gun which shoots bullets will probably be a hitscan weapon (unless you want bullet drop). There is no special setup needed to create a hitscan weapon.

Launcher: Launcher weapons, when fired, spawn a game object into the world. An example of a projectile weapon is a rocket launcher which shoots rockets that move across the screen. The actual behavior of the projectile must be handled with scripts on the actual projectile - so the rocket fired from the rocket launcher would need the code that makes it go forwards, and the code that makes it explode and deal damage, attached. This means, however, that you can make a projectile weapon which can do nearly anything! A projectile weapon has no special setup.

Spray: Spray weapons are weapons which deal damage in a constant area while the fire button is held down. The primary use for the spray setting is flamethrowers and flamethrower like weapons. In order to function, spray weapons require a game object with a particle emitter and a script called "SprayScript" to be a child of the spray weapon. Damage is calculated when the particles emitted from this object collide with things in the scene.

If you want a visual effect separate from the particles which actually cause damage and collide, you can attach additional objects with particle emitters as children of the object with SprayScript. All these emitters will be activated/deactivated automatically. In fact, you may decide to make the damaging particles totally invisible!

Melee: Melee weapons cause damage by colliding with a target. Melee weapons must have a collider with the HitBox script attached. The Hitbox has a reference to the GunScript for the weapon and controlls the damage, and  force. The collider  The rate of attack, the attack length and the recovery time are set in GunScript. Using Bullet Penetration

You may want to have some guns be able to shoot through some walls, obstacles, or other surfaces. The FPS Constructor has a simple system for setting this up in your scenes. First, you need to determine what objects you want bullets to be able to penetrate. In FPSContructor/ScriptsDB/CommonlyUsed, you will find a script called "BulletPenetration". This script must be attached to every object you want any bullets to be able to pass through.

The script allows you to customize the level of resistance to bullets the material will have. Every hitscan weapon has a variable under the "Shot Properties" foldout called "Penetration Level". Similarly, every object with the BulletPenetration script has a "Penetrate Value". If the gun's Penetrate Value is greater than or equal to the shot object's Penetration Level, then the bullet will keep going. Furthermore, when a bullet goes through an object, its Penetrate value is reduced - so a shot from a gun with a Penetration value of 2 could pass through two objects with a Penetration Level of 1, but could not pass through a third.

There is no need to manually attach the BulletPenetration scripts to the objects in your scene since it it managed by the Effects and Penetration Wizard described in the Demo Scene section of this document.

Secondary Fire

 You may want to create a gun which has an alternate fire or firing mode. Doing so is simple. The first step is to attach a second copy of GunScript to the weapon you want to have the additional firing mode. In the Inspector, open the foldout labeled "Alt-Fire Properties". There is a checkbox labeled "Primary Weapon". This should be checked on the copy of GunScript which describes the weapon's primary fire, and unchecked for the secondary fire.

There are two ways in which alternate fires can function. The first is where the player has to switch to and from alt-fire mode - this is similar to weapon attachments in games such as Call of Duty. The second is where the player can instantly fire either the primary or secondary weapon - such as in the Unreal Tournament series. You can select either of these modes by checking or unchecking the box labeled "Instant Fire" in the "Alt-Fire Properties" foldout. Note that this option is found only on the secondary weapon's gunscript. If unchecked, the player will have to switch between primary and secondary fire modes.

If you leave the box unchecked, there will be a few other variables you to deal with. "Enter Secondary Time" and "Exit Secondary Time" determine how long, in seconds, it takes to switch to and from the secondary fire mode, respectively. Secondary Interrupt, if checked, allows the player to cancel the weapon's firing animation to switch modes.

Beyond this, simply edit the second copy of GunScript as normal.

Tracers

 Tracers are a feature of hitscan weapons where some shots are actually drawn on the screen. This can be useful for giving players a visual indicator of where they are shooting - or where they are being shot at from. Note that only hitscan weapons can use tracers - only hitscan weapons could need them.

   

To use tracers, you first need to create what you want the tracer to look like. A tracer will take the form of a game object with a particle system on it. You can find an example system on the FAL in the demo scene. . When you've made this object, make it a child of the gun's AimObject.

Once you have your tracer attached, open the "Audio + Visual" tab of the GunScript. There are three fields: "Tracer", "Shots Per Tracer", and "Simulate Tracer For". The first field is easy - simply drag the tracer object you attached into the field.

You may not want every shot to have a tracer (with a fast-firing weapon, for example). "Shots Per Tracer" allows you to determine how often a tracer is drawn. If this is set to 1, every shot will be a tracer; if it is set to 2, every other shot; if it is set to 3, every third shot, and so on. A value of 0 means that no tracers will appear.

You tracers will not look right if you do not tweak the final field, "Simulate Tracer For". We have to attach the tracer object to the AimObject so that we can project the tracer in the direction the gun fired. But because the AimObject is virtually always going to be behind the barrel of the gun, just drawing the tracer from the AimObject would make the tracer look like it is coming from behind the player's gun. "Simulate Tracer For" allows you to simulate the tracer for a little bit before actually drawing it - the larger this value is, the longer the tracer will be invisible. Just tweak this until you find a value that looks good.

Shared Ammo

You may want two or more guns to use the same type of ammunition. If you have a stock AK-47 and an AK-47 with a scope, for example, it makes no sense for both guns to have their own supply of ammo!

To designate that a weapon shares ammo, open the "Ammo + Reloading" tab in the GunScript and check the variable labeled "Shares Ammo". When this is checked, a several new variables should appear.

The first of these, "Ammo Set Used:" determines which pool of ammo the weapon will actually draw from. The package supports up to ten different ammo pools (called Ammo Sets). By default these should be named Ammo Set 1, Ammo Set 2, etc. To keep things straight, you may want to change the name in the next field - "Rename Ammo Set".

Now, of course, you need to do this for any additional guns you want to share ammo with this gun, making sure they all use the same Ammo Set. Note that if you any of the ammo settings for one gun (clips, max clips, infinite ammo), the values will change for any other gun using that ammo set.

Shell Ejection

 Shell ejection requires two things- a shell prefab and a GameObject. The weapon will automatically instantiate the shell at the GameObject's position. All you need to do is activate the 'Shell Ejection" checkbox on the GunScript and drag in the prefab and GameObject. There is a default shell prefab included in PrefabsDB/Extra.

Tuning Accuracy + Recoil

 The Weapon Accuracy and recoil is controlled by setting the Accuracy Properties on the GunScript. These properties give you full control over the accuracy of the weapon. Tool Tips are provided for all values.

Damage Falloff

Bullets lose velocity (and power) the farther they travel. Flames from a flamethrower, too, become weaker the farther from the source they move. Alternatively, some fictional weapons may get stronger the farther they have traveled. By default, weapons you create in the package will have the same damage and force up close as they do at maximum range. You can change this by using damage falloff. In Spray weapons, damage falloff is always enabled. If you want to enable it in a Hitscan weapon, you'll need to open the "Shot Properties" tab in the GunScript and check the box labeled "Use Damage Falloff".

When you enable damage falloff, a number of additional fields will appear. These fields are the same in both Hitbox and Spray types.

  • Damage at max falloff:  Read-only field that shows you how much damage the weapon currently does at max range, so you don't have to do the math.
  • Force at Max Falloff: Read-only field that shows the force at max range.
  • Min Falloff Distance: If you hit anything closer than this distance you will do full damage and force.
  • Max Falloff Distance: Beyond this distance force and damage of the weapon won't change
  • Damage Coefficient: For every unit of distance traveled past the Min Falloff Distance, the weapon's damage is multiplied by this value until the Max Falloff Distance is reached.
  • Fallof Distance Scale: Defines how many meters count as one "unit of distance" when calculating falloff. Using a small scale means that your weapon will have a smooth damage curve; larger values will mean the weapon's damage will rise or fall in sudden bursts.

Charge Weapons

You may want to design a weapon which can "charge up" if the user holds down fire, such as the Covenant Plasma Pistol in Halo. Projectile charge weapons can also be used to create some unique effects - for an example, see the Lock-On-Missiles and Orbital Laser examples in the Demo Scene.

Both Hitscan and Projectile weapons can be charge weapons, although they must be configured differently. The variables which can be affect by a Hitscan weapon's charge level are hardcoded in - charging a Hitscan weapon can change specific things. In contrast to this, the charge level of a projectile weapon is passed to the projectile object when it is spawned, and it is up to you to program the behavior you want it to exhibit. It is also possible to make additional scripts for hitscan weapons which read the charge level and change values in the GunScript or other places.

Both Gun Types share the basic setup in common. First, under the Fire Properties foldout, check the box labeled Charge Weapon. Once checked, Minimum Charge and Maximum Charge should appear. Internally, the weapon keeps track of a "Charge Level". This is simply the amount of time (in seconds) that the player has been charging the weapon by holding down the fire button. If the player lets go of the fire button and their Charge Level is less than the Minimum Charge, the weapon won't fire. The player's charge level will never be greater than the Maximum Charge.

Must Fire When Charged does just what it says - when this setting is enabled, the gun will automatically fire if it reaches the maximum charge. If this isn't enabled, the player can keep the weapon charged until they decide to fire.

There are also a few settings in other parts of the GunScript that will appear if your weapon is a charge weapon. In the Audio + Visual foldout, a field labeled Charging Sound will appear. You have the option of dragging in a sound which will be looped while the weapon is charging. In the Ammo + Reloading foldout, an option labeled Ammo Cost Per Charge will appear. This allows you to make charged shots cost more ammunition. When the player fires, the value is multiplied by the charge level, and the result is rounded down to a whole number (you can't shoot half a bullet!). This is then subtracted from the player's ammo IN ADDITION to the normal cost for firing. You can't charge up a shot so that firing it would cost more ammo than you have - you can only charge up to what you can afford. If you have Must Fire When Charged enabled, the weapon will automatically fire if you charge the weapon up to the point that firing it would use up all your ammo.

One thing to keep in mind is that charge weapons are designed to be very flexible, and that you  will have to write a script to make the most interesting charge weapons (see the Orbitial Laser, Seeker Missiles, and Plasma Cannon as examples) by taking advantage of the fact that GunScript's variables are public and can be modified/read by your charge script.

Configuring the Rest of Your Game

Making Things Take Damage

    What good is making all these weapons if you can't shoot anything with them? Thankfully, making characters and objects in your scene take damage is very simple. In the folder FPSConstructor/ScriptsDB/CommonlyUsed, you will find a script named EnemyDamageReceiver. If you want something to take damage, attach this script to it - it doesn't have to be an "enemy."

You'll also set the health of the object in this script, by changing the variable Hit Points.

When the object is killed/destroyed, you also have the option of instantiating an effect and spawning a replacement object. Death Effect is an object which is spawned upon the enemy's death. This can technically be anything; an example use might be an explosion when destroying a robotic enemy. You can delay the effect occurring (in case you have an animation you want to play first, for example) by entering the desired delay in seconds into the Effect Delay field. These effects are optional.

Finally, you may want to spawn a new object directly where the destroyed object was. An obvious example is replacing a soldier enemy with a "dead" ragdoll version, so that the enemy does not simply disappear on death. Simply drag the object you would like to be spawned into the field labeled Dead Replacement.

You don't have to worry about the Player Object field - the object will find the player on start by the tag "Player". Still, you can save some load time if there are many objects using EnemyDamageReceiver by hand-dragging the player object into this field.

If Use Hit Effect is enabled, every time the object takes damage it will display a small indicator on the player's HUD and play a sound. This is useful because it lets the player know that their attacks are hitting! If your game has many non-player sources of damage, and you still want to use hit indicators, you may have to edit this script. Otherwise, the player will see the indicator even when they are not the one doing the damage. One possible solution is to use a separate function to apply damage from NPCs.

This script is very lightweight, and is very edit friendly! You can change the way your objects or enemies receive damage and react to it by modifying this script - you don't need to change any other scripts. Just make sure to keep using a function named "Apply Damage" which takes a float!

In this script you'll find a few variables which can be changed. The Hit Texture is what the player will actually see when the hit effect appears. Bullet Decals, Impact Effects, and Footsteps

    When you shoot a wooden wall with a high-powered rifle, you expect it to leave a mark. You might also expect shooting wood to make a different sound than shooting concrete, and you might expect splinters to fly from the wood but not from the concrete. When you walk on grass it should sound different than walking on tile. You can use the FPS Constructor's effects systems to easily include all these details in your scene.

    To begin with, you need to have a Manager object in your scene. If you don't have one in your scene already, you can find a prefab, called Manager, in FPSConstructor/PrefabsDB/Setup. Drag this into your scene (preferably the top level).

Now you'll need to create an "effects set." Each effects set will include decals, bullet sounds, impact effects, and footstep sounds; you can make any object use this effects set. You can make as many of these sets as you want (the default limit is 75, which is exceedingly high, but you can technically increase this). You may want one generic set, or specific sets for every material type. To create a set, select the manager object in your scene, and, in the attached Effects Manager script, select the drop-down for "Select Set to Edit" and select New Set. By default, your set will be named "Set 0." You can rename it by entering a new name in the Effects Set Name field.

Each set contains the ability to customize a number of things. Every field is optional - if you don't put in any bullet decals, for example, the package won't complain. For each field, there should be one blank GameObject slot. Every time you drag an object in, however, the list will expand. The actual effect applied will be selected randomly from the list. Here's a breakdown of what each effect is:

Bullet Decals: Bullet decals are applied to a surface when a hitscan weapon hits. They are generally bullet holes. A bullet decal should take the form of a thin square or plane with a transparent texture on it. You can find example bullet decals in FPSConstructor/PrefabsDB/Extra/Decals. The maximum number of decals in the world is set at the top of the script, in the field labeled Maximum Decals in World. A high number will have an effect on performance.

You can manually apply a decal by calling the function ApplyDecal in the EffectsManager script. ApplyDent takes an array as an argument; the array holds, in order:

hit.point, hitRotation, hit.transform, hit.normal, hitSet where

  • hit is a raycast hit
  • hitSet is the effect set used by the object being hit.
  • hitRotation is equal to: Quaternion.FromToRotation(Vector3.up, hit.normal) where "hit" is that same raycast hit.

Dent Decals: Dent decals are simply alternate decals for different types of weapons. Melee weapons, for example, will use dent decals instead of bullet decals. You may also want to make some projectile weapons use these decals instead. You can manually apply them by calling the function ApplyDent in the EffectsManager script. It takes the same argument as the ApplyDecal function detailed above.

Hit Particles: Hit particles are particles which are emitted where a bullet hits. These could be sparks if the attack hits metal, or splinters of wood if the attack hits wood. A hit particle takes the form of a Game Object with a particle system on it; the particle system should be a one-shot system.

You can manually instantiate hit particles by calling the function ApplyHitEffect in the EffectsManager script. It takes the same argument as the ApplyDecal function detailed above.

Bullet Sounds: These are sounds played when a bullet hits a surface. This should take the form of an audio clip. You can manually call a bullet sound by calling the function BulletSound in the EffectsManager script. It takes the same argument as the ApplyDecal function detailed above.

Collision Sounds: Much like how dent decals are an alternative to bullet decals, collision sounds are alternatives to bullet sounds. Melee weapons will naturally use collision sounds instead of bullet sounds.  You can manually call a collision sound by calling the function CollsionSound in the EffectsManager script. It takes the same argument as the ApplyDecal function detailed above.

Footstep Sounds: Footstep sounds are played automatically when the player walks on a surface using a decal set with sounds. You can control the frequency of the footstep sounds, as well as the volume, in the script called Footstep, which is attached directly to the player. Footstep Interval determines the time interval in between footsteps; Footstep Volume determines the volume the footstep will be played at.

Once you have set up one or more effects sets, you need to determine what surfaces will use effects. Every object that you want to use any effects must have the script UseEffects, which can be found in FPSConstructor/ScriptsDB/CommonlyUsed. You can then set what effects set that object will use from a dropdown menu.

 

For each object there is a checkbox and a dropdown menu. If the checkbox is checked, the UseEffects script will be attached to all colliders with that name; if it is unchecked, the script will be removed if it is attached. If the box is checked you can also use the dropdown to pick what effects set will be used for objects with that name.

Friend/Foe Colors for Crosshairs

   It is possible to easily make the default crosshairs change color when the player is aiming at specific objects. For example, you could make the crosshairs turn green when aiming at a friendly target, or make them turn yellow when you are aiming at an object the player can interact with.

There is a script called CrosshairColor which can be found in FPSConstructor/ScriptsDB/CommonlyUsed. This script should be attached to anything you want to change the player's crosshair color. There are three groups - Friend, Foe, and Other. In the dropdown you can pick which one you want this object to use.

Next, you'll need to set what color/texture each of those three groups will use. Select the Weapon Camera in the hierarchy view (it should be directly attached to the Player object). You should see a script called Smart Crosshair. In the inspector for this script there are three texture fields that you can use to pick the texture for each crosshair color group. They are Friend Texture, Foe Texture, and Other Texture. The crosshair's texture will become this texture when hovering over a object in the corresponding group. If you leave this empty, you can also make the crosshair disappear.

Customizing Your Control Scheme (Version 1.1+)

In the 1.1 update there is a new input system which allows you to easily configure your own control schemes without editing the core FPS Scripts. This is especially important now that the system supports mobile platforms. 

This is done using the new InputDB system, in which the InputManager object has objects for each input. Each input object has a script called InputItem which holds the information about that input, and which you can manipulate through your own scripts. The InputItem components are by default found under the InputManager object under Manager.

The default scripts ('Key' and 'Axis') simply implement the standard keyboard controls, but your own scripts can do anything you want. The key is that the InputItem script has four values, each corresponding to a standard Input variable or state used by the system :

  • up : GetButtonUp (true for the first frame a key is released)
  • down : GetButtonDown (true for the first frame a key is pressed)
  • got : GetButton (true as long as the key is pressed)
  • axis : corresponds to axis values (floats)

Through editing how these values are set on each Input object, you can make any control scheme you want. The only special case is for weapon switching, which requires you to change a static value called WeaponSelector.selectedWeapon to be the index of the weapon you want to select, in addition to indicating a button press.

The Mobile Demo (Version 1.1+)

Now that the FPS Constructor supports mobile devices, we added a demo scene showing an example of how to implement mobile controls. It is not meant to be a perfect control scheme, and the buttons are currently placed only for an iPad, but is just an example of how you can use the new Input system to create a mobile control scheme.

We decided to create controls for the following Inputs for our example, with explanations of what each input does and how we decided to implement them on mobile.

Buttons: Each button had to control the up, down, and got values of their InputItem. SwitchWeapon also had to change the static WeaponSelector.selectedWeapon. 

  • Fire1 - fires weapons - touch button on screen

Axes: Each axis had to change the axis value of their InputItem. The variable 'mobile' on LockCursor had to be set to true to allow this to work.

  • Vertical - forward/backward movement of player - tilt control

You can look at the components on these inputs (under the InputManager object). This example shows how open-ended the new input system is, allowing you to control inputs in any way you want.

Controlling the Player (Version 1.1+)

Version 1.1 adds several static boolean variables and functions that you can use to control the player from your own scripts. This can be used for any number of things, from preventing the player from firing during a cutscene to making the player climb over obstacles like in Call of Duty.

The variables are the following:

  • PlayerWeapons.canLook - can the player move the camera?
  • PlayerWeapons.canSprint - can the player sprint?
  • PlayerWeapons.canMove - can the player move?
  • PlayerWeapons.canFire - can the player fire their weapon?
  • PlayerWeapons.canAim - can the player aim?
  • PlayerWeapons.canCrouch - can the player crouch?
  • PlayerWeapons.doesIdle - do idle animations activate?
  • PlayerWeapons.canInteract - can the player interact using the 'interact' key?
  • PlayerWeapons.CanSwitchWeapon - can the player switch weapons?

There are also several static functions that you can use to control if the weapon is shown on screen:

  • PlayerWeapons.HideWeapon() - makes the player hide their weapon, using a put away animation if there is one
  • PlayerWeapons.HideWeaponInstant() - makes the player's weapon instantly disappear
  • PlayerWeapons.ShowWeapon() - makes the player take out their weapon

These variables are never changed by the system, so you have complete control of their state.
